让 Codex 接管思源笔记，并打通 Hexo 博客仓库

把 AI、笔记和代码仓库接成一条工作流之后，AI 才不只是“会聊天”，而是开始具备可执行的工作能力：先到思源里找方案和草稿，再回到本地仓库改代码、写文章，最后把结果再回写到思源。

这次打通了什么

  • Codex 现在可以通过全局 siyuan MCP 直接操作思源笔记
  • 思源默认走本机 http://127.0.0.1:6806
  • 本机思源程序路径固定为 D:\siyuan\SiYuan.exe
  • 运行中的思源 workspace 位于 C:\Users\24160\SiYuan
  • 额外补了一个全局 skill，用来把思源、当前工作区和博客仓库串起来

现在 AI 可以直接做的事

  • 列出和搜索思源笔记本、文档和标签
  • 读取笔记内容，整理成任务说明或博客草稿
  • 新建、追加、覆盖、移动思源文档
  • 在批量操作前创建快照
  • 先读思源里的方案，再改本地仓库
  • 把仓库中的修改结果回写到思源

这套链路是怎么工作的

1. 用户级全局接入思源 MCP

思源 MCP 不再只绑定当前博客仓库，而是挂到了用户级 C:\Users\24160\.codex\config.toml。这样无论在哪个项目里启动 Codex，只要是这个用户，都可以复用同一套思源接入能力。

2. 本地启动脚本负责兜底

如果 127.0.0.1:6806 没启动，启动脚本会尝试拉起本机思源，再等待接口就绪。这样平时不用反复手动检查服务是否在跑。

3. 思源 skill 负责把笔记和仓库串起来

我额外补了一个全局 skill，专门处理以下场景：

  • 先搜索思源里的方案，再改代码
  • 按思源草稿生成博客文章
  • 把这次修改总结回写到思源
  • 在博客仓库 d:\blog\butterfly 下优先读取 AI_OPERATION_GUIDE.md

推荐工作流

  1. 先让 AI 搜索思源里相关笔记
  2. 确认范围后，让 AI 开始改仓库
  3. 如果是大批量整理，先做思源快照
  4. 改完后，把结果同步回思源
  5. 博客内容确认后，再执行 hexo generate 检查生成产物

适合拿来做什么

  • 把思源里的任务笔记直接转成可执行开发任务
  • 把思源草稿整理成博客文章并落进 Hexo 仓库
  • 把调试记录、配置说明、部署结果统一回写到思源
  • 让笔记、代码和产出不再分裂在三个地方

当前落地结果

这次已经完成了三件事：

  • 思源 MCP 改成了全局可用
  • 全局 skill 已经补齐
  • 当前博客仓库已经纳入这套联动流程

后续在 Codex 里直接说下面这类话就可以：

  • “先搜索我思源里关于这个功能的笔记，再改仓库”
  • “读取思源里的博客草稿，整理成 Hexo 文章”
  • “把这次修改结果回写到思源”
  • “先给思源做快照，再批量整理标签或文档”

小结

当 AI 能同时看见笔记、工作区和博客仓库时，它才真正从“对话工具”变成“执行工具”。接通思源之后，很多原本要手工在多个软件之间来回搬运的事情，终于可以被收拢成一条更顺手的工作链路。
